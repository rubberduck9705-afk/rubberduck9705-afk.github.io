SQL-Injection in search parameter :

Application is running on LAMP-Stack : Linux, Apache, MySQL & PHP

Overview :

This is a SQL-injection attack targeting a vulnerable search feature/parameter in a “fake” banking application.

This show why inadequate input sanitization is needed.

In this case as u will see the vulnerability lies in the vulnerable parameter: search inside transactions.php. The issue exists in the way user input is directly incorporated into SQL queries without proper validation or parameterization.

First we will display how to exfiltrate data from the database manually and then utilize SQLmap to dump the entire db.

How It Works Manually :

Closing the Original Query :

')

The application has the following query :

$search = $_GET['search'] ?? '';

if($search) {
    $query = "SELECT * FROM transactions WHERE user_id = $user_id AND (description LIKE '%$search%' OR recipient LIKE '%$search%') ORDER BY transaction_date DESC";
  • The ') closes the original LIKE clause and its parentheses.
  • This effectively terminates the legitimate query.

First part of the attack is always to identify whenever SQLi is possible and how the backend handles user supplied input.

By using a automated tool like Burp Suite we can use a brute attack like Intruder (Sniper Attack) or in this case do it manually :

' -> Throws a 500

Its a question of trying different things and payloads to identify whats working :

')-- -

Payload above worked.

The Simple Explanation :

The original query has :

LIKE '%$search%'

When we inject ')-- - it becomes :

LIKE '%')-- -

' - Closes the string that started with '%

) - Closes the parenthesis from (description LIKE ... OR recipient LIKE...)

-- - SQL comment symbol (Everything after this is ignored/commented out)

Basically it says : “End the search here ), then run MY query UNION SELECT Then ignore everything else -- -

UNION SELECT statements :

Using the above statement. Whenever a webapplication returns the results of the query within it HTTP response it is possible to use UNION to get/retrive data from other tables within the applications database.

Before we can take advantage of this we have to determine the number of columns that are being returned within the original query.

For this we will utilize the ORDER BY clause and keep increaing the column index until the application throws an error. The error indicates that the number of tables are invalid/incorrect.

Payload : 

') ORDER BY 1-- -

In this case we can just inject the payload “as is” without any URL-encoding since our browser handles this automatically “In this instance” :

We increase the index/value until the application throws/returns an error :

') ORDER BY 8-- -

Going beyond the value of 8 we receive an error :

Therefor, we know the application returns 8 columns and we can proceed to our UNION Attack :

For this, we will submit a series of NULL values matching the numer of columns(8) within the table :

') UNION SELECT NULL, NULL, NULL, NULL, NULL, NULL, NULL, NULL-- -

To determine the datatype of a column (integer or string), we can simply make the NULL value into a string by using ''. Again, if the application returns an error we know that it isn’t valid or correct and by doing it methodically for each NULL value we can enumerate the data type of each column within the table.

') UNION SELECT 'NULL', NULL, NULL, NULL, NULL, NULL, NULL, NULL-- -

For the sake of our sanity most of the NULL values are strings which makes os proceed to the next step. Getting the table name :

Using a common set of views called information_schema.tables we can list the tables in the database :

') UNION SELECT table_name, NULL, NULL, NULL, NULL, NULL, NULL, NULL FROM information_schema.tables-- -

Data is obviously returned by the output/values are being shown in the wrong column. Remember that each NULL value is also a column within the table and depending of the structure if the page it isn’t shown probably. Hence why we try and move the table_name to another column/position :

The output now shows us a lot of different tables. In this case we will focus on extracting/dumping the data from the users table :

Now we can do the exact same thing with the column names but this time focusing on the table users :

') UNION SELECT NULL, NULL, column_name, NULL, NULL, NULL, NULL, NULL FROM information_schema.columns WHERE table_name='users'-- -

In the payload about we changed it to → Get the names of each column WHERE table_name = ‘users’ :

The juicy part is now, to use concatenation to extract the contents of the columns username and password :

') UNION SELECT NULL, NULL, CONCAT(username,':',password), NULL, NULL, NULL, NULL, NULL FROM users-- -

CONCAT() is a SQL function that combines multiple strings into one.

Without CONCAT() we would only be able to extract one column at a time.

CONCAT() :

  • Combine multiple columns into one visible field
  • Add separators (like : or | to tell where one value ends and another begins
  • Extract more data per query

We succesfully dumped the users credentials. Note that in a real case, these would most likely be encrypted with a strong algorithme. Basically the database would hold the hash of each users password and not in plain text as in this example.

Bonus : SQLMap :

The manual workflow can be automated using SQLMap.

To run the attack with SQLMap we first have to give it the following information :

  • -u = Specifies the target URL
  • --cookie flag = Sends cookies with each request
  • --batch = SQLMap automatically chooses deffault answers (carefully use this).
  • --dump = Extract and display ALL data from discovered databases

Command : 

sqlmap -u "http://192.168.126.128/securebank/transactions.php?search=test" \
       --cookie="PHPSESSID=q3k2if2q8hdb31ir5i6pairpcn" \
       --batch \
       --dump
